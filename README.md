# Task Manager


## App Description

Create, edit, complete, delete, and search personal tasks. Every task has a title,
an optional description, and a completion flag. Tasks are stored locally with Hive
and persist across app restarts.

**Core features implemented**
- Task list with an empty state (both "no tasks yet" and "no search results")
- Add / edit task (title required, description optional)
- Toggle completion from the list (checkbox) without opening the edit screen
- Delete with a confirmation dialog (swipe-to-delete or the trailing delete icon —
  both routes require confirmation)
- Local persistence via Hive
- Search bar that filters by title or description, live as you type

## Architecture



```
lib/
├── core/
│   ├── constants/        # box name, typeId
│   └── theme/
└── features/
    ├── splash/presentation/splash_screen.dart
    └── tasks/
        ├── models/task.dart                     # HiveObject + manual TypeAdapter
        ├── repository/
        │   ├── task_repository.dart              # abstract contract
        │   └── task_repository_impl.dart         # the only class that imports package:hive
        └── presentation/
            ├── providers/task_providers.dart     # box → repository → Notifier + derived providers
            ├── screens/                          # TaskListScreen, AddEditTaskScreen
            └── widgets/                          # TaskTile, ProgressRing, UpNextCard, ...
```

`Task` (`lib/features/tasks/models/task.dart`) is both the thing the UI displays
and the thing Hive stores — `extends HiveObject` gives it `.save()`/`.delete()`
directly, so `TaskRepositoryImpl` can mutate a task's fields and call `.save()`
instead of re-serializing it.

**Why a repository, and why it's small:** `TaskNotifier` never imports
`package:hive` — it only calls methods on `TaskRepository` (`getTasks`,
`addTask`, `updateTask`, `toggleCompletion`, `deleteTask`). `TaskRepositoryImpl`
is the one class in the app that knows a `Box` exists. That buys two concrete
things: storage can be swapped later (SQflite, a REST API) by writing a new
`TaskRepository` implementation and changing one provider override, with zero
changes to the notifier or any screen; and the notifier's logic can be tested
against a fake in-memory repository instead of real Hive I/O (see
`test/task_notifier_fake_test.dart`). What it deliberately *doesn't* have is a
separate domain entity, a `Result<T>`/`Failure` error type, or `AsyncNotifier`
— those added layers without adding flexibility, see Challenges below.

## State Management — Riverpod


- `taskBoxProvider` is overridden once in `main.dart`, after `Hive.openBox`
  resolves.
- `taskRepositoryProvider` watches the box and constructs `TaskRepositoryImpl`
  — this is the seam: override this provider instead of `taskBoxProvider` to
  swap storage entirely.
- `TaskNotifier` (`task_providers.dart`) depends only on `taskRepositoryProvider`.
  `build()` returns the initial state — like a Cubit's constructor. Each method
  (`addTask`, `updateTask`, `toggleCompletion`, `deleteTask`) calls the
  repository, then reassigns `state` — like `emit(...)`. Every widget watching
  `taskProvider` (or anything derived from it) rebuilds automatically.
- `searchQueryProvider` is a plain `StateProvider<String>`, kept separate so
  typing never touches storage.
- `filteredTasksProvider`, `taskStatsProvider` (progress ring numbers), and
  `upNextTaskProvider` (hero card) are all just `Provider`s that `ref.watch`
  the task list and derive something from it — no manual "recompute and
  notify" wiring needed, Riverpod's dependency graph handles that.


## Persistence — Hive

`Task` is a `HiveObject` with a **manually written** `TypeAdapter` rather than
one generated by `build_runner`/`hive_generator`. This was a deliberate choice —
see Challenges below.

## Challenges & Decisions

- **Codegen vs. manual adapters:** Hive's usual workflow generates the
  `TypeAdapter` via `build_runner`. I wrote it by hand instead so the project
  builds with a plain `flutter pub get` and no generation step, which is one less
  thing that can go wrong for anyone cloning the repo on a machine with a slightly
  different toolchain. The trade-off is a few more lines of boilerplate in
  `task.dart`.
- **Repository interface, but no `Result<T>`:** an earlier pass had a full
  domain/data/presentation split with a `Result<T>`/`Failure` error wrapper
  threaded through every layer. The repository split earns its keep — it's
  what makes `TaskNotifier` testable without real storage, and what would let
  storage change later without touching the UI. The `Result<T>` wrapper
  didn't earn the same keep, so it's gone: repository methods either return
  normally or throw, and the one or two call sites that can fail (saving,
  deleting, from the UI) catch with a plain `try`/`catch`.
- **Delete confirmation on two paths:** the spec requires a confirmation dialog
  before deletion. Since `Dismissible` (used for swipe-to-delete) and the trailing
  delete icon are two different gestures, both route through the same
  `_confirmDelete` dialog so there's only one place that owns that copy/logic.


## Running the project

This repo was authored as pure Dart/Flutter source (no Flutter SDK was
available in the environment it was written in), so it's missing the native
`android/`, `ios/`, and `web/` platform folders that `flutter create` normally
generates. **Run this once** after unzipping, before anything else:

```bash
flutter create . --project-name task_manager --org com.example.taskmanager
```

This scaffolds the missing platform folders in place without touching
`lib/`, `test/`, `pubspec.yaml`, or `README.md`. Then:

```bash
flutter pub get
flutter run
```

No `build_runner` step is required — the Hive adapter is hand-written and
registered directly in `main.dart`.

To produce a release APK:

```bash
flutter build apk --release
# Output: build/app/outputs/flutter-apk/app-release.apk
```
